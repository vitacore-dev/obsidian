## Symptoms

- Один или несколько SQL-запросов зависли или выполняются слишком долго.
- Сессия блокирует ресурсы и влияет на работу приложения.
- Требуется быстро остановить проблемный backend-процесс PostgreSQL.

## Scope

- Версия продукта: `all`
- Окружение: любое, где используется PostgreSQL
- Точка применения: поддержка/администрирование БД

## Diagnosis

1. Подключиться к PostgreSQL с правами на просмотр активности.
2. Получить список активных процессов:
   ```sql
   SELECT *
   FROM pg_stat_activity
   WHERE state = 'active';
   ```
3. Найти `pid` процесса, который нужно остановить.
4. Убедиться, что это действительно проблемная сессия (по пользователю, запросу, времени выполнения).

## Resolution / Workaround

1. Отменить запрос по `pid`:
   ```sql
   SELECT pg_cancel_backend(<pid>);
   ```
2. Проверить, что сессия/запрос больше не активны в `pg_stat_activity`.
3. Если запрос не отменяется, эскалировать на DBA/инженера эксплуатации для принудительного завершения через `pg_terminate_backend`.

## Escalation

Эскалировать, если:

- проблемный процесс не останавливается через `pg_cancel_backend`;
- блокировки продолжают влиять на критичный функционал;
- повторяемость инцидента указывает на дефект в приложении или схеме.

Эскалировать в:

- Команда: `DBA / Engineering On-call`
- Передать: `pid`, текст запроса, пользователь БД, время начала запроса, влияние на клиентов
