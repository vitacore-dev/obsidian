# Обзор проекта: база знаний техподдержки (Obsidian + GitHub)

## 1. Цель

Проект внедряет единую базу знаний технической поддержки, где Obsidian используется для написания материалов, а GitHub — для совместной работы и контроля качества.

Ключевые цели:

- сократить среднее время решения обращений;
- сделать знания по диагностике и устранению проблем переиспользуемыми и легко находимыми;
- стандартизировать создание, ревью и обновление статей.

## 2. Операционная модель

Проект построен по принципу docs-as-code:

- инженеры поддержки редактируют Markdown-заметки в Obsidian;
- общая часть хранится в Git;
- все изменения проходят через pull request в `main`;
- ревьюеры проверяют структуру, корректность и практическую применимость.

Это обеспечивает:

- прозрачную историю изменений;
- понятную ответственность за разделы;
- обязательные quality-gates перед публикацией.

## 3. Структура репозитория

Репозиторий разделен по рабочим сценариям поддержки:

- `01_Runbooks` — пошаговые инструкции диагностики и восстановления;
- `02_Known-Issues` — повторяющиеся проблемы, причины и обходные решения;
- `03_Product-FAQ` — стандартизированные ответы на частые вопросы клиентов;
- `04_Integrations` — материалы по интеграциям и ограничениям (кроме выделенных разделов по СУБД);
- `05_Incidents-Postmortem` — разборы инцидентов и выводы;
- `06_Onboarding` — обучение и адаптация новых сотрудников;
- `07_MS-SQL` — Microsoft SQL Server: runbook и справочные заметки для поддержки;
- `08_PostgreSQL` — PostgreSQL: runbook, конфигурация и инструменты для поддержки;
- `90_Templates` — шаблоны заметок.

Ключевые файлы управления процессом:

- `CONTRIBUTING.md` — регламент работы с контентом;
- `.github/pull_request_template.md` — обязательный шаблон PR;
- `CODEOWNERS` — владельцы разделов и правила ревью;
- `README.md` — быстрый старт и ежедневный цикл работы.

## 4. Процесс совместной работы

Ежедневный цикл для автора:

1. Синхронизация локальной копии:
   - `git checkout main`
   - `git pull --rebase`
2. Создание рабочей ветки:
   - `kb/add-<topic>` для новых материалов;
   - `kb/fix-<topic>` для обновлений.
3. Редактирование заметок в Obsidian.
4. Коммит и push ветки.
5. Открытие PR в `main`.
6. Получение комментариев и доработка при необходимости.
7. Merge после апрува.

Режимы ревью:

- Fast lane — небольшие изменения (опечатки, локальные правки, уточнения);
- Standard lane — новые runbook, изменения процесса, критичный контент.

## 5. Стандарты качества контента

Обязательные поля в YAML frontmatter:

- `title`
- `owner`
- `last_updated`
- `product_version`
- `tags`

Обязательные разделы по типу заметки:

- Runbook (`01_Runbooks`)
  - Symptoms
  - Scope
  - Diagnosis
  - Resolution / Workaround
  - Escalation
- Known Issue (`02_Known-Issues`)
  - Issue Summary
  - Symptoms
  - Root Cause
  - Workaround
  - Permanent Fix Status
- Product FAQ (`03_Product-FAQ`)
  - Question
  - Short Answer
  - Steps for Customer

Эти ограничения гарантируют, что статья пригодна для работы в реальной смене поддержки.

## 6. Реализованная автоматизация

### 6.1 Проверки GitHub Actions

Файл: `.github/workflows/kb-checks.yml`

Что делает:

- запускается на `push` и `pull_request` в `main`;
- поднимает Python 3.11;
- запускает скрипт валидации базы знаний.

Технический порядок выполнения в CI:

1. GitHub получает событие (`push` или `pull_request`) для `main`.
2. Запускается job `validate-kb` на `ubuntu-latest`.
3. Шаг `actions/checkout@v4` загружает код репозитория в runner.
4. Шаг `actions/setup-python@v5` устанавливает Python `3.11`.
5. Шаг `python scripts/validate-kb.py` выполняет проверки структуры и метаданных.
6. По коду возврата скрипта GitHub определяет статус:
   - `0` -> проверка успешна (green check);
   - `1` -> проверка не пройдена (red check), PR требует исправлений.

Практический эффект:

- ревьюеры получают уже предварительно проверенный контент;
- снижается количество ручных замечаний по формальным требованиям;
- повышается единообразие материалов в ключевых разделах базы знаний.

### 6.2 Скрипт валидации

Файл: `scripts/validate-kb.py`

Логика проверки:

- проверяет наличие frontmatter;
- проверяет обязательные поля метаданных;
- проверяет обязательные заголовки разделов в зависимости от папки;
- завершает CI с понятными ошибками по каждому файлу при нарушениях.

Детализация логики работы:

1. **Выбор файлов для проверки**
   - скрипт сканирует каталоги `01_Runbooks`, `02_Known-Issues`, `03_Product-FAQ`, а также `07_MS-SQL` и `08_PostgreSQL`;
   - в каталогах `07_MS-SQL` и `08_PostgreSQL` обязательные секции runbook применяются только к файлам с именем, начинающимся с `RUNBOOK`;
   - внутри каждого из перечисленных каталогов проверяются все файлы `*.md`.
2. **Проверка YAML frontmatter**
   - файл должен начинаться с блока `--- ... ---`;
   - при отсутствии блока добавляется ошибка `missing yaml frontmatter`.
3. **Проверка обязательных полей**
   - для frontmatter проверяются поля: `title`, `owner`, `last_updated`, `product_version`, `tags`;
   - если поле отсутствует или пустое, добавляется ошибка вида `missing frontmatter field: <field>`.
4. **Проверка структуры разделов**
   - набор обязательных секций определяется по имени папки;
   - если в тексте нет обязательного заголовка, добавляется ошибка `missing section: <section>`.
5. **Формирование результата**
   - ошибки выводятся по каждому файлу в явном формате;
   - если есть хотя бы одна ошибка, скрипт завершает работу с кодом `1`;
   - если ошибок нет, печатается сообщение об успешной проверке и возвращается код `0`.

Пример ожидаемого вывода при ошибке:

- `[ERROR] 01_Runbooks/<file>.md`
  - `missing frontmatter field: owner`
  - `missing section: ## Escalation`

Пример ожидаемого вывода при успехе:

- `Validation passed for N file(s).`

### 6.3 Промпт для легковесной AI-проверки PR

Файл: `docs/PR_BOT_PROMPT.md`

Поведение модели:

- ревьюит измененные документы поддержки;
- отмечает недостающие разделы и нечеткие инструкции;
- отмечает потенциально чувствительные данные;
- возвращает унифицированный ответ: `approve` или `request_changes`, критичные проблемы, улучшения, краткий вывод.

## 7. Стартовый контент

В репозитории уже добавлены примеры, которые задают стандарт:

- runbook по ошибке входа через SSO timeout;
- known issue по дублирующимся уведомлениям после reconnect;
- FAQ по корректному сбору логов для поддержки.

Эти статьи можно использовать как эталоны при создании новых материалов.

## 8. Безопасность и гигиена данных

Проект ориентирован на исключение чувствительных данных из базы знаний.

Текущие меры контроля:

- правила в `CONTRIBUTING.md`;
- подтверждение в PR checklist, что чувствительные данные не добавлены;
- автоматическая структурная валидация в CI;
- правила `.gitignore` для локальных артефактов Obsidian и файлов плагинов.

## 9. Ожидаемые результаты

Операционные эффекты внедрения:

- ускорение триажа и решения типовых обращений;
- более стабильное качество диагностики на L1/L2;
- снижение дублирования поиска решений;
- упрощение онбординга новых сотрудников;
- прозрачное и непрерывное улучшение базы знаний.

## 10. Рекомендуемые следующие шаги

Ближайшая итерация:

1. Заменить плейсхолдеры в `CODEOWNERS` на реальные GitHub-команды/пользователей.
2. Добавить в CI проверку ссылок и сканирование на секреты.
3. Добавить ежемесячную проверку устаревших материалов (по `last_updated`).
4. Подключить автоматические комментарии PR-бота на выбранной легковесной модели.
5. Зафиксировать KPI:
   - время от решения тикета до публикации статьи;
   - доля PR, принятых с первого ревью;
   - доля тикетов, закрытых с использованием базы знаний.
