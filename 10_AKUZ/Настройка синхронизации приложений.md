Система ЛЛО объединяет работу ИС «АПС», ИС «РДС» и модуль «ЛЛО» в ИС «АКУЗ».

Для корректной работы всей системы необходимо полное совпадение справочников и структур данных в трех приложениях. Синхронизация приложений происходит по запросам и автоматической передаче справочников и регистров из ИС «РДС» в ИС «АПС» и ИС «АКУЗ». Обмен данными происходит посредством службы RabbitMQ.

## 1. Раздел `<appSettings>`

Необходимо указать следующие параметры:

| Параметр               | Значение                                                                             | Описание                                                                                                                                 |
| ---------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `RDSDictionariesQueue` | `rds.dictionary`                                                                     | Наименование очереди в RabbitMQ для синхронизации справочников по ЛЛО (для массового запроса справочников в ИС «АКУЗ»)                   |
| `RDSRegistryQueue`     | `rds.registry`                                                                       | Наименование очереди в RabbitMQ для отправки в RDS запросов, связанных с регистрами по ЛЛО (для массового запроса регистров в ИС «АКУЗ») |
| `AMPQ`                 | `server=0.0.0.0; queue=queue; User=user; Password=password; process=RDSLoadProcess;` | Параметры настройки службы RabbitMQ                                                                                                      |

### Состав параметра `AMPQ`:
- **server** – IP-адрес сервера, где расположена служба RabbitMQ
- **queue** – наименование прослушиваемой очереди, из неё AIS LPU забирает пакеты
- **User** и **Password** – логин и пароль в RabbitMQ
- **process** – команда (для модуля ЛЛО — `RDSLoadProcess`)

**Пример:**
```xml
<add key="RDSDictionariesQueue" value="rds.dictionary" />
<add key="RDSRegistryQueue" value="rds.registry" />
<add key="AMPQ" value="server=0.0.0.0; queue=queue; User=user; Password=password; process=RDSLoadProcess;" />
```
## 2. Раздел `<AKUZ.Server>`

Необходимо указать следующие параметры:

|Параметр|Значение|Описание|
|---|---|---|
|`AMPQConnectionString`|`server=0.0.0.0; user=user; password=password`|IP-адрес расположения службы RabbitMQ и учетные данные для доступа|
|`AMPQEnabled`|`true`|Включение работы с открытым протоколом передачи сообщений AMQP (Advanced Message Queuing Protocol), платформа RabbitMQ|
|`ServerName`|`name`|Наименование, по которому ИС «РДС» определяет, куда отправить ответ по пришедшему из ИС «АКУЗ» пакету. В RabbitMQ должна быть настроена очередь с таким же наименованием, прослушка очереди в конфигурационном файле ИС «АКУЗ», сервер синхронизации в ИС «РДС» с таким же наименованием|
|`QueueName`|`name`|Наименование очереди, в которую отправляются пакеты для RDS. Если не указаны отдельные очереди для `RDSDictionariesQueue` и `RDSRegistryQueue`, запросы будут отправлены в указанную здесь очередь|
|`ChildBenefitCode`|`000`|Код льготы для категории «Дети до трех лет». При создании нового пациента в возрасте до 3-х лет будет автоматически формироваться направление с указанной льготой. **Для отключения функционала** — указать несуществующий код льготы|

**Пример:**

```
<AMPQConnectionString value="server=0.0.0.0; user=user; password=password" />
<AMPQEnabled value="true" />
<ServerName value="name" />
<QueueName value="name" />
<ChildBenefitCode value="000" />
```

### Ошибки синхронизации

В папке **«Синхронизация приложений/Ошибки»** содержатся все ошибки, которые возникают при передаче информации в ИС «РДС» от других программ. В ошибке указаны:
- Тип пакета, по которому пришла ошибка
- Очередь, в которую был отправлен пакет
- Описание ошибки
![[Pasted image 20260507135742.png]]

#### Порядок обработки ошибок

1. Если ошибка связана с перебоями в связи или с несовпадением справочников, то после восстановления связи и синхронизации справочников необходимо **повторно обработать ошибку**.
 ![[Pasted image 20260507135752.png]]
2. Если после повторной обработки ошибка дублируется (у записей совпадает значение атрибута «Идентификатор сообщения»), значит не были внесены нужные изменения.
3. Если ошибка связана с некорректными данными, переданными в пакете, то необходимо **исправить данные в той системе, из которой пришел пакет** (ИС «АПС», ИС «АКУЗ»), пакет автоматически будет отправлен в ИС «РДС».
4. Успешно обработанные ошибки **удалить из списка**.
 ![[Pasted image 20260507135814.png]]

### Серверы синхронизации

МО и АО, по которым передается информация между информационными системами, указываются в **серверах синхронизации**.

|---|---|
| **Наименование** | Должно совпадать с наименованием очереди в RabbitMQ |
| **Тип адреса** | `Queue` |
| **Тип сервера** | `APS` — для аптечных организаций<br>`AKUZ` — для МО |

> **ВНИМАНИЕ!!!** Необходимо добавить **все МО и АО**, которые участвуют в обмене, иначе информация передаваться не будет.

# 3. Настройка службы RabbitMQ

Для синхронизации приложений (ИС «АПС», ИС «РДС» и ИС «АКУЗ» модуль «ЛЛО») в RabbitMQ необходимо завести очереди, посредством которых будет происходить обмен данными (см. Руководство администратора RabbitMQ).

## Примеры разграничения очередей

| Назначение очереди                          | Требование к наименованию                                                                                             |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **Очередь для справочников ЛЛО**            | Должно совпадать с очередью, указанной в конфигурационном файле ИС «АКУЗ»                                             |
| **Очередь для регистров ЛЛО**               | Должно совпадать с очередью, указанной в конфигурационном файле ИС «АКУЗ»                                             |
| **Очередь для рецептов**                    | Должно совпадать с названием очереди, указанной в ИС «АКУЗ» в действиях при изменении объектов для `RECIPE_SIGNATURE` |
| **Очередь для заявок**                      | Должно совпадать с названием очереди, указанной в ИС «АКУЗ» в действиях при изменении объектов для `DRUG_REQUEST`     |
| **Очередь для получения пакетов ИС «АКУЗ»** | Должно совпадать с наименованием, указанным в конфигурационном файле сервера приложений ИС «АКУЗ»                     |
| **Очередь для получения пакетов ИС «РДС»**  | Должно совпадать с наименованием, указанным в конфигурационном файле сервера приложений ИС «РДС»                      |
| **Очередь для получения пакетов ИС «АПС»**  | Должно совпадать с наименованием, указанным в конфигурационном файле сервера приложений ИС «APS»                      |

> **Примечание:** Очереди могут быть созданы как отдельные для каждого действия, так и объединены.
