# Инструкция: подключение нового Obsidian-инстанса к базе знаний (Windows)

Документ описывает полный путь для Windows: от установки инструментов до первого Pull Request.

## 1. Предварительные условия

У сотрудника должны быть:

- доступ в GitHub-организацию;
- доступ к репозиторию `vitacore-dev/obsidian`;
- права на создание веток и PR.

## 2. Установка инструментов на Windows

Установить:

- Obsidian (desktop, Windows);
- Git for Windows;
- опционально: GitHub CLI (`gh`) или GitHub Desktop.

Проверка в `PowerShell`:

- `git --version` должен возвращать версию Git;
- опционально: `gh --version` должен возвращать версию GitHub CLI.

## 3. Клонирование репозитория (отдельно и подробно)

Этот шаг делается один раз на рабочем ноутбуке. Ниже безопасная последовательность, чтобы избежать ошибок с путями и правами.

### 3.1. Открыть PowerShell и перейти в рабочую директорию

1. Открыть `PowerShell` (обычный, не обязательно от администратора).
2. Перейти в папку, где будет храниться репозиторий, например:
   - `cd $HOME\Documents`
3. Проверить текущий путь:
   - `Get-Location`

Рекомендация: не клонировать в системные каталоги (`C:\Windows`, `C:\Program Files`) и в очень длинные вложенные пути.

### 3.2. Проверить, нет ли уже папки `obsidian`

Перед клонированием проверить содержимое текущей директории:

- `Get-ChildItem`

Если папка `obsidian` уже существует:

- если это старый/битый клон - удалить или переименовать папку;
- если это рабочий клон - не делать повторный `git clone`, а перейти в папку и выполнить `git pull --rebase`.

### 3.3. Выполнить клонирование

Основной вариант (HTTPS):

- `git clone https://github.com/vitacore-dev/obsidian.git`

Альтернативный вариант (SSH, если уже настроен ключ):

- `git clone git@github.com:vitacore-dev/obsidian.git`

После успешного клонирования перейти в репозиторий:

- `cd .\obsidian`

### 3.4. Проверить, что клон корректный

Внутри папки репозитория выполнить:

- `git remote -v` - проверить, что есть `origin` с правильным URL;
- `git branch --show-current` - обычно должна быть ветка `main`;
- `git status` - должно быть сообщение, что рабочее дерево чистое.

Ожидаемый результат: репозиторий подключен, ветка корректна, локальных изменений нет.

### 3.5. Частые ошибки при клонировании и что делать

- **`Repository not found`**
  - проверить URL и доступы в GitHub-организации.
- **`Authentication failed` / запрос логина бесконечно**
  - пройти авторизацию через `gh auth login` или GitHub Desktop, затем повторить clone.
- **`destination path 'obsidian' already exists and is not an empty directory`**
  - перейти в существующую папку и работать с ней, либо удалить/переименовать папку и повторить clone.
- **Клонирование прошло, но нет доступа к push**
  - проверить, что выданы права на запись в репозиторий (не только read).

## 4. Подключение vault в Obsidian

1. Открыть Obsidian.
2. Выбрать **Open folder as vault**.
3. Указать локальную папку репозитория `obsidian`.
4. Дождаться индексации файлов.

## 5. Настройка плагина Obsidian Git

1. Открыть `Settings -> Community plugins`.
2. Убедиться, что включен `Obsidian Git`.
3. Проверить доступность команд в `Ctrl + P`:
   - `Obsidian Git: Pull`
   - `Obsidian Git: Commit`
   - `Obsidian Git: Push`
   - `Obsidian Git: Open source control view`

Рекомендованные настройки:

- использовать ручные `Pull/Commit/Push` (без агрессивной авто-синхронизации);
- перед началом работы всегда выполнять `Pull`;
- писать понятные сообщения коммитов.

## 6. Первый запуск: синхронизация

1. В Obsidian выполнить:
   - `Obsidian Git: Pull`
2. Убедиться, что нет ошибок аутентификации.
3. Открыть любую заметку и проверить, что контент читается корректно.

Если pull не проходит:

- проверить, что вход в GitHub выполнен (через credential manager/GitHub Desktop/`gh auth login`);
- проверить URL репозитория;
- выполнить `git pull` в `PowerShell` и посмотреть текст ошибки.

## 7. Ежедневный рабочий цикл (обязательный)

Перед задачей:

1. `git checkout main`
2. `git pull --rebase`
3. `git checkout -b kb/add-<topic>` или `kb/fix-<topic>`

Во время задачи:

1. Редактировать заметки в Obsidian.
2. Проверить изменения в `Obsidian Git: Open source control view`.
3. Сделать commit с понятным сообщением.

После задачи:

1. Выполнить `Obsidian Git: Push` (или `git push -u origin <branch>` при первом push).
2. Открыть PR в `main`.
3. Пройти ревью и выполнить merge.

## 8. Первый учебный PR (чеклист)

1. Создать ветку `kb/fix-onboarding-<name>`.
2. Сделать небольшое улучшение в одной заметке.
3. Запустить локальную проверку:
   - `python scripts/validate-kb.py`
4. Commit -> Push -> PR.
5. Исправить замечания ревью и смержить.

## 9. Типичные ошибки и быстрые решения

- **Ошибка аутентификации GitHub**
  - Выполнить повторную авторизацию (`gh auth login` или вход через GitHub Desktop).
- **Конфликт при pull**
  - Не коммитить конфликтные файлы без проверки; запросить помощь ревьюера/лида.
- **В PR попали лишние локальные файлы**
  - Проверить `.gitignore`, особенно `.obsidian/*`.
- **CI не прошел валидацию**
  - Проверить обязательные поля frontmatter и структуру заметки.

## 10. Что прочитать после подключения

- `README.md`
- `CONTRIBUTING.md`
- `docs/PROJECT_OVERVIEW.md`
- `06_Onboarding/Onboarding Checklist.md`

## 11. Критерий успешного подключения

Сотрудник считается подключенным, если он:

- успешно выполняет Pull/Commit/Push из Obsidian Git;
- создает рабочую ветку и открывает PR в `main`;
- проходит локальную и CI-валидацию без критичных замечаний;
- соблюдает правила работы с чувствительными данными.
