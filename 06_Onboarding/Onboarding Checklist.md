# Чеклист подключения нового сотрудника к базе знаний

## 1. Настройка доступов

- Получить доступ в GitHub-организацию и репозиторий.
- Быть добавленным в команды поддержки для ревью и владения разделами.
- Проверить, что доступны ветки, PR и настройки репозитория.

## 2. Подготовка локального окружения

1. Установить необходимые инструменты:
   - Obsidian
   - Git
   - Опционально: GitHub CLI (`gh`)
2. Клонировать репозиторий локально.
3. Открыть папку репозитория как vault в Obsidian.
4. Включить плагин `Obsidian Git`.

## 3. Базовые материалы и процесс

- Прочитать `README.md`.
- Прочитать `CONTRIBUTING.md`.
- Прочитать `docs/PROJECT_OVERVIEW.md`.
- Пройти `06_Onboarding/Инструкция: подключение нового Obsidian-инстанса.md`.
- Изучить структуру каталогов:
  - `01_Runbooks`
  - `02_Known-Issues`
  - `03_Product-FAQ`
  - `90_Templates`

## 4. Стандарты контента

Каждая операционная заметка должна содержать поля frontmatter:

- `title`
- `owner`
- `last_updated`
- `product_version`
- `tags`

Перед написанием использовать соответствующий шаблон:

- `90_Templates/Runbook Template.md`
- `90_Templates/Known Issue Template.md`

Никогда не добавлять:

- пароли, токены, секреты;
- персональные данные клиентов;
- конфиденциальные артефакты клиента без обезличивания.

## 5. Первый учебный Pull Request (обязательно)

1. Синхронизировать локальную ветку:
   - `git checkout main`
   - `git pull --rebase`
2. Создать учебную ветку:
   - `git checkout -b kb/fix-onboarding-<name>`
3. Сделать небольшое полезное изменение:
   - улучшить формулировку в FAQ, или
   - добавить одну перекрестную ссылку, или
   - уточнить один шаг в runbook.
4. Запустить локальную валидацию:
   - `python3 scripts/validate-kb.py`
5. Сделать commit и push ветки.
6. Открыть pull request в `main`.
7. Обработать комментарии ревью и выполнить merge.

## 6. Ежедневный рабочий цикл

Перед каждой задачей:

- `git checkout main`
- `git pull --rebase`
- создать ветку `kb/add-...` или `kb/fix-...`

После каждой задачи:

- запустить скрипт валидации;
- сделать commit с понятным сообщением;
- выполнить push и открыть PR;
- после merge вернуться в `main` и подтянуть последние изменения.

## 7. Ожидания на первые две недели

- Использовать шаблоны для всех новых заметок.
- Писать коротко, прикладно и воспроизводимо.
- Добавлять условия эскалации в runbook с высоким влиянием.
- Запрашивать ранний фидбек по первым 2-3 PR.

## 8. Критерии завершения

Онбординг считается завершенным, когда сотрудник может:

- создавать и обновлять заметки в правильных разделах;
- проходить проверки локально и в CI;
- открывать и доводить PR до merge с минимальными доработками;
- стабильно соблюдать правила гигиены данных.
