## Что показывает система

- Статусы по XML-файлам (request/reply/unknown).
- Нормализованные статусы обработки СМЭВ.
- Цепочки по `MessageID`: отправка, доставка, ответы, дубли, SLA.
- Журнал событий с деталями по файлам.

## Базовые действия оператора

1. Нажать «Обойти папки сейчас» для немедленного полного скана.
2. Проверить путь каталога XML и при необходимости применить новый путь.
3. При накоплении мусорных записей очистить `journal.json` через UI-кнопку.
4. Для точечной очистки удалить отдельную запись из таблицы.
5. При диагностике открыть полный XML через кнопку `xml`.

## Работа с MessageID

- Использовать фильтры направления (`Все`, `Только нам`, `Только от нас`) и роли.
- Раскрывать цепочку для просмотра последовательности событий.
- Проверять SLA-метрики (`S->D`, `S->R`, `S->Last`) для оценки задержек.

## Частые наблюдения

- `доставлено > отправлено` — обычно дубли reply-событий.
- `sent = 0`, но события есть — исходный `SendRequest` отсутствует в текущем журнале.
- Нет `messageData` — часть транспортных ответов приходит без бизнес-данных.

## Рекомендуемый сценарий диагностики

1. Проверить блок аномалий.
2. Отфильтровать поток (нам/от нас).
3. Найти проблемный `MessageID`.
4. Раскрыть цепочку и оценить этап, на котором застряло сообщение.
5. При необходимости открыть XML и передать детали в эскалацию.
