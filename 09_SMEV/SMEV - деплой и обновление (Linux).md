## Назначение

Краткая операционная инструкция по запуску, обновлению и проверке работоспособности SMEV Viewer на Linux.

## Предусловия

- Node.js 18+ и npm установлены.
- Есть доступ к каталогу с исходниками.
- Подготовлены конфиги: `.env`, `monitor-config.json`, `adapter-config.json`.
- У пользователя сервиса есть права на каталоги входящих XML и директорию данных.

## Где скачать проект

- GitHub-репозиторий: `https://github.com/vitacore-dev/SMEV_viwer`
- Клонирование:
  - `git clone https://github.com/vitacore-dev/SMEV_viwer.git`

## Первичный запуск

1. Перейти в `packages/smev-folder-monitor`.
2. Установить зависимости:
   - `npm install --workspaces=false`
3. Собрать проект:
   - `npm run build`
4. Запустить backend:
   - `SMEV_MONITOR_PORT=3847 node dist/server/index.js`
5. Проверить доступность API и UI.

## Обновление версии

1. Остановить сервис (systemd/pm2/manual).
2. Обновить код из репозитория.
3. Повторно выполнить `npm install --workspaces=false`.
4. Выполнить `npm run build`.
5. Запустить сервис.
6. Проверить:
   - чтение директорий мониторинга;
   - обновление `journal.json`;
   - отображение цепочек по `MessageID`.

## Быстрые проверки после деплоя

- Процесс слушает ожидаемый порт.
- В UI появляются новые файлы после скана.
- Нет ошибок парсинга XML в логах.
- Журнал не поврежден и пополняется новыми записями.

## Типовые проблемы

- Сервис не стартует: конфликт порта или ошибка в `.env`.
- Нет новых событий: неверные пути мониторинга или права доступа.
- Пустой UI: backend работает, но не читает `journal.json`/каталоги.

## Эскалация

Эскалировать в `engineering-integrations`, если:

- повторяемые ошибки парсинга на валидных XML;
- неконсистентные цепочки `MessageID` после обновления;
- деградация производительности при штатной нагрузке.
